Data Sources

How the Account Research Tool collects, verifies, and applies data from every research source.

Two Generation Pipelines

Both pipelines are accessible from the Account Research menu in the bookscrub sheet and produce identical 9-section output documents. They share the same upstream inputs and the same downstream DocGenerator.gs — the difference is entirely in how the LLM is called and what knowledge it can reach.

Original Pipeline (v1)

Generate for Company…
  • OpenAI GPT-4o via INFRA /openai
  • Optional Bing web search grounding (gb=1)
  • Public web knowledge only
  • ~2 min · stable · fully operational

Glean Pipeline (v2)

Generate via Glean…
  • Gemini via Glean agent + INFRA /glean
  • Glean internal index: CRM, deal notes, account history
  • External web search included in every run
  • ~3 min · coming online · richer context
Source 1 — Internal Bookscrub Sheet

The Docusign Book of Business export — a Google Sheet with one row per account, containing subscription, consumption, and product data sourced from Salesforce and billing systems. This is the only required input; no data preparation needed before running the tool.

Identity & Contract

Account name, Salesforce ID, GTM group, plan tier, ARR/ACV, contract dates, seats purchased and active

Consumption

Envelopes sent vs. purchased, utilization rate, CLM and IAM usage metrics per account row

Product Signals

15 products + 4 bundles scored by generateProductSignals() — injected as ground truth the LLM cannot override

GTM group mode is supported: accounts sharing a Salesforce group ID are aggregated by getGtmGroupData(), summing ACV, envelopes, and seats across all members before the LLM payload is assembled.

Sources 2–4 — Public Enrichment APIs

Before any LLM call is made, DataEnricher.gs pre-fetches verified company facts from three free public APIs. The data flows through both pipelines and is injected into LLM prompts as authoritative anchors. SEC financial data is additionally enforced post-response — enforceVerifiedData() overwrites any conflicting AI estimates before anything is written to the document.

Public API Sources
SEC EDGAR (via Cloudflare Worker) Revenue, COGS, OpEx, CapEx, net income, employee count, and business segment revenue parsed from 10-K XBRL filings. A Cloudflare Worker proxies the SEC API to bypass GAS CORS restrictions. CIK resolved via Wikidata P5531.
Wikipedia REST API 2–3 sentence company overview for the Company Profile section. The matched article title provides a reliable bridge to the correct Wikidata entity — more accurate than a raw Wikidata text search, which can return subsidiaries or unrelated entities.
Wikidata API CEO (P169), headquarters (P159), founding date (P571), stock ticker (P249/P414 qualifier), SEC CIK (P5531), and industry (P452). The CIK from P5531 is the key that connects a company name to its EDGAR filing record.
Lookup chain Company name → Wikipedia search → article title → pageprops → Wikidata QID → entity claims → SEC CIK → Cloudflare Worker → EDGAR XBRL. Each step degrades gracefully — a miss at any point returns empty fields.
Sources 5–6 — INFRA Proxy Endpoints

Both LLM pipelines route through infra.agreementsdemo.com — an internal Docusign proxy that authenticates to the underlying AI services on behalf of the tool. GAS never communicates with OpenAI or Glean directly, and no account data leaves the corporate environment. Auth uses DOCU-INFRA-IC-KEY and DOCU-INFRA-IC-USER headers configured as GAS script properties.

/openai — Original Pipeline

Model: GPT-4o · gb=0 primary, gb=1 fallback
  • Call 1 — Account Profile: overview, BUs, SWOT, executives, tech stack
  • Call 2 — Business Map: org hierarchy with agreement intensity
  • Call 3 — Agreement Landscape: top 20 types scored by volume + complexity
  • Call 4 — Commerce Estimate: revenue potential by dept and agreement type
  • Call 5 — Priority Map: ranked opportunities + 90-day action plan

/glean — Glean Pipeline

Gemini via Glean agent · 5-step workflow
  • Steps 1+2 (parallel) — company-search (internal index) + web-search
  • Step 3 think1 — Company Profile synthesis from both research inputs
  • Step 4 think2 — Business Map, Agreement Landscape, Commerce Estimate
  • Step 5 think3 — Docusign strategy briefing + Big Bets
  • Steps 1+2 use fetchAll for true parallelism; steps 3–5 sequential
Resilience: Every call is wrapped in try/catch — a failed call returns {} and the affected section renders a fallback. /openai Call 3 has a simplified-prompt retry; double-failure falls back to generateFallbackAgreementLandscape(), a deterministic result from industry config tables in Config.gs. /glean think steps retry 3× on HTTP 500/502 with a 60s wait between attempts. Research steps (1+2) are non-fatal — think steps degrade gracefully with empty strings.
Data Assembly & Flow

Inputs

  • Bookscrub sheet — identity, contract, usage
  • Product signals — 15 products + 4 bundles scored
  • Wikipedia — company overview text
  • Wikidata — CEO, HQ, ticker, CIK
  • SEC EDGAR — verified financials (via CF Worker)
→

LLM Pipeline

  • Original: 5 sequential calls → /openai
  • Glean: 2 parallel + 3 sequential → /glean
  • SEC data enforced post-response
  • Citation markers stripped (cleanCitations)
  • JSON parsed + normalised (tryParseJson)
→

Output Document

  • DocGenerator.gs — 9 section builders
  • Sections 1–5, 7–9: internal + LLM
  • Section 6: internal data only (no LLM)
  • Sources appendix: deduplicated + filtered
  • Saved to configured Google Drive folder
